## 2. Notion Developer Platform 3.5 — Workers, External Agents API, and the `ntn` CLI

**Why it's trending:** Notion launched its Developer Platform 3.5 on **May 13, 2026** — the most significant developer-facing release in Notion's history. Three core primitives shipped simultaneously: (1) **Notion Workers** (Beta) — a TypeScript/Node.js hosted runtime (`ntn workers deploy`) that lets developers write custom code for syncing external data into Notion databases, building agent tools callable by Notion Custom Agents, and receiving webhooks from external services — all without managing servers. Workers run in a secure sandbox on Notion's infrastructure, free during beta, transitioning to Notion credits August 11 2026. (2) **External Agents API** (Alpha) — lets teams bring external AI agents (Claude, Codex, Decagon, custom-built) into Notion as orchestration targets; a Decagon support ticket can auto-route to a coding agent, which proposes a fix and loops a human in for approval, all within Notion. (3) **Notion CLI** (`curl -fsSL https://ntn.dev | bash`) — purpose-built for developers and coding agents, covering workspace auth, read/write actions, Workers build/deploy, and MCP support. Notion's MCP server also updated: now supports Meeting Notes and block comments, with database creation/update 91% more token-efficient. Notion confirmed a Developer Platform Hackathon for May 16–17 2026. InfoWorld covered May 14. Official YouTube tutorials published May 13 (two videos). The r/Notion subreddit flagged the launch May 13. The founder explicitly positioned this as "any data, any tool, any agent, all running on our infrastructure" — making Notion an application development platform, not just a productivity tool.

**Suggested blog angle:** *"Notion Workers: I synced Salesforce, GitHub, and Stripe into Notion in one afternoon — here's how it works"* — a developer-first tutorial covering: what Workers actually are vs. Zapier/Make (the key architectural difference: deterministic TypeScript vs. LLM reasoning, cheaper, more reliable for defined input/output tasks), the five-minute setup (install `ntn` CLI, scaffold a project with `ntn workers create`, write a sync in `src/index.ts`, push secrets, deploy), a live worked example syncing GitHub PR data into a Notion database (the actual TypeScript code, what the Worker object looks like, how `worker.sync()` works, what the deploy flow is), a second example building a custom agent tool (write a tool that creates a Jira ticket when a Notion agent says "escalate this" — using `worker.tool("createJiraTicket", {...})` and `context.notion` for the Notion API client), the webhook pattern (how `worker.webhook("onGithubPush", {...})` lets any app trigger Notion — close a task when a PR merges, update CRM when a subscription changes), the cost model (free during beta → Notion credits August 11 — what that means per sync run), and two things that are still rough (alpha External Agents API requires waitlist, OAuth setup order matters: env push before oauth start or you get a confusing error). Alternative angle: *"I built a custom Notion agent tool with the new Workers CLI — here's the whole process from zero to deployed."*

**Competition notes:**
- InfoWorld (May 14): news/analyst framing — "Notion courts developers" — 3-minute news read, no tutorial.
- Notion official release notes (notion.com/releases/2026-05-13): the primary changelog — comprehensive but not a tutorial.
- Notion official YouTube (May 13): two videos — "How to Build Agent Tools on Notion's Developer Platform" and "How to Orchestrate Any Agent on Notion's Developer Platform" — official demos, not independent developer narratives.
- Notion official docs (developers.notion.com/workers): thorough reference docs and guide for writing an agent tool — primary authority for API-level queries, but dry reference format.
- sintesi.studio (May 10): "Notion Workers: Build custom agent tools or sync any data source into Notion" — a genuine hands-on post (the Spotify playlist sync example with OAuth code) — the strongest independent post so far, published 3 days before the official launch. Good, but covers only one specific use case (Spotify) and doesn't address the agent tools pattern or webhook pattern.
- r/Notion (May 13): community posts flagging the launch — awareness only.
- Reddit "I tested Notion Workers for the first time" (Apr 3): an early alpha tester's walkthrough of a Spotify sync — predates the official 3.5 launch, covers an older version of the Workers API, but establishes that hands-on content resonates with the Notion developer community.
- **No independent developer has published a post covering the complete Workers 3.5 workflow: sync + agent tools + webhooks + CLI setup + cost model**, in the context of the official May 13 launch.
- Search intent: how-to + evaluation. Audience: developers who use Notion and want to extend it programmatically; product/eng teams evaluating whether to replace Zapier/Make with native Workers; developers building Notion agents who want to give their Custom Agent access to external APIs; indie builders using Claude Code who want to integrate Notion into their agent workflow.
- **Low competition** on "Notion Workers tutorial," "Notion Workers CLI setup," "ntn workers deploy guide," "Notion Developer Platform 3.5 walkthrough," or "build Notion agent tool Workers."

The Hackathon (May 16–17) will drive a new wave of builders looking for tutorials. The "replace Zapier with native Notion Workers" angle is extremely sticky for Notion-native teams.

## 3. GitHub Copilot → AI Credits Migration — The Survival Guide Before June 1

**Why it's trending:** GitHub announced on **April 27, 2026** that all Copilot plans will switch from flat-rate Premium Request Units (PRUs) to **GitHub AI Credits** (token-based billing) on **June 1, 2026**. The community response has been explosive: 571 comments and 679 replies on the GitHub community FAQ thread; a viral Reddit post titled "The new Copilot pricing makes zero sense — why am I paying $39/mo for $39 in expiring API credits?" attracting developer backlash; Visual Studio Magazine publishing "Devs Sound Off on Usage-Based Copilot Pricing Change: 'You Will Get Less, but Pay the Same Price'" (April 27); and Yahoo Tech: "Developers face new, chaotic Copilot pricing" (May 1). On **May 12, 2026**, GitHub released April usage reports so developers can preview what their June bills will look like — creating a concrete, time-sensitive action for every Copilot user in the next 2 weeks. Key developer pain points: (a) Opus 4.6 via Copilot will cost 27× more per request than Sonnet 4.6 under the new multipliers; (b) code review via Copilot now consumes GitHub Actions minutes on top of AI Credits; (c) three separate metrics (monthly PRU budget, weekly token budget, overage budget) are shown in completely different places with no unified view; (d) unused credits don't roll over; (e) annual plan subscribers face an especially confusing transition timeline. One developer in the community thread documented the three-metric confusion in detail, noting that "the most visible metric (the % in VS Code) is actually the least relevant one for most users." Meanwhile, the May 12 GitHub changelog update explicitly instructs users to download the April usage report as a CSV to model their June costs before it's too late.

**Suggested blog angle:** *"GitHub Copilot AI Credits: how to download your April report, read the numbers, and decide what to do before June 1"* — a practical, step-by-step guide covering: the five-minute download flow (billing overview → Preview my bill → Download CSV), what the `aic_quantity` and `aic_gross_amount` columns actually mean and how to identify your top-cost surfaces (Chat vs. code review vs. cloud agent), the three-tier model confusion decoded (monthly credits ≠ weekly token budget ≠ overage credits — what each one controls and where to find each setting), the Opus multiplier math (how to calculate how many Opus vs. Sonnet interactions your credits actually buy, and whether the mix is right for your workflow), the decision framework for each plan type (monthly Pro/Pro+ → migrate automatically Jun 1; annual plan → do nothing or convert early for prorated credits; Business/Enterprise → promotional buffer runs Jun–Aug), the four realistic alternatives for developers who conclude Copilot is now too expensive for their agentic use (Claude Code Max plan, Cursor $20/month with pooled models, direct API via Cline with LM Studio local or OpenRouter pay-as-you-go, or the Copilot Auto Mode as a cost-saving default), and the one setting everyone should check today (budget cap for additional spend — currently defaults to $0 which is safe, but can be changed). Alternative angle: *"I modeled my Copilot costs under the new AI Credits system — here's whether it's still worth $39/month."*

- Visual Studio Magazine (Apr 27): strong news coverage, developer sentiment focus — no setup walkthrough or cost modeling.
- The Decoder (Apr 28): news summary — no actionable guide.
- Thurrott.com (Apr 27): news/commentary — no tutorial.

- **No independent developer has published a structured "download your report, interpret the numbers, decide what to do" guide** that covers the April report release (May 12), the three-metric confusion, the Opus multiplier math, and a clear decision tree for each plan type.

Audience: every Copilot user who got the April billing report email (May 12), especially Pro+ users doing agentic work who are about to face a cost shock, engineering managers who need to advise their teams, and individual developers choosing between staying on Copilot vs. switching to Claude Code or Cursor.
